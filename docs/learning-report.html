Learning Report · Architecture & Code Flow

How Rewind works, from keystroke to disk

Rewind is a local-first API client — a Postman built as an Electron app where nothing leaves the machine. This report traces the real code paths end to end so you can open any file and know exactly where you are.

Electron · main / preload / renderer React 18 + Zustand TypeScript, electron-vite ~60 source files
00 — START HERE

The mental model

Before any file, hold these five ideas in your head. Almost every design decision in the codebase falls out of one of them.

◆ The five load-bearing ideas

  • Two worlds, one bridge. The UI (renderer) is a sandboxed web page that cannot touch the disk or network. All privileged work happens in the Electron main process, reached only through a typed window.relay.* bridge.
  • Edits live in memory as “drafts.” Typing in the editor never writes to disk. It updates a drafts[requestId] override. Disk is touched only on explicit Save or on structural changes.
  • Two-phase variable resolution. The UI previews {{vars}} with dynamic values left literal; the main process does the real substitution at send time — so what you see is what gets sent.
  • Local-first, crash-safe storage. Everything is JSON/JSONL under Electron’s userData, written atomically (temp → fsync → rename) with corrupt-file quarantine.
  • The app used to be called “Relay.” You’ll see window.relay, .relay bundles, and relay:* keys everywhere — kept deliberately for data compatibility. Mentally read “relay” as “the bridge.”

The rest of this report is a guided descent: start at the process boundary, work down through the data model and state, then follow a single POST from the moment you press Send all the way to a line appended in a run log — and back up to the response you see.

01 — THE SHAPE OF THE APP

The security posture is textbook-hardened Electron, set in window.ts:35: contextIsolation: true, nodeIntegration: false, sandbox: true, webSecurity: true. Navigation is frozen — will-navigate is always prevented and new windows open in the OS browser instead. The renderer is welded to its own document; the relay preload API is its entire outside world.

▸ Trace it yourself

Startup order lives in src/main/index.ts. Note the hard sequence in whenReady: initProfiles() must run before anything reads a file, because it sets the active-profile directory that every path helper depends on. Miss that and paths.ts throws "profile dir not initialized".

02 — THE ONLY DOOR

The IPC boundary

Because the renderer can’t act on its own, every capability is a named IPC channel. The channel names are a single frozen source of truth in src/shared/ipc.ts, imported by both sides so a typo can’t silently drift the two apart.

The preload layer (src/preload/index.ts) wraps each channel in a one-line typed method and exposes them as window.relay via contextBridge.exposeInMainWorld. The renderer’s index.d.ts augments the global Window so you get full type-safety with zero runtime coupling:

// preload — every method is this thin
send: (payload) => ipcRenderer.invoke(IPC.httpSend, payload),
// two subscriptions return an unsubscribe fn (removeListener, not removeAll)
onRunAppended: (cb) => { /* subscribe to push channel */ },

The channel catalogue

All channels are request/response (ipcMain.handle) except two pushes from main. Registration is in src/main/ipc/register.ts.

Notice the guards: shell:openExternal and the window’s new-window handler both refuse anything that isn’t http(s), closing off file:/javascript: abuse.

03 — THE VOCABULARY

The data model

Everything in src/shared/types.ts is a pure type (plus two runtime exports). Learn this hierarchy and the rest of the app reads like prose.

flowchart TD
    WS["Workspace
id · name · activeEnvironmentId"] --> COL["Collection
name · version · variables?"]
    COL --> ITEMS["items: TreeNode[]"]
    ITEMS --> FOLDER["FolderNode
children: TreeNode[]"]
    ITEMS --> REQ["RequestNode"]
    FOLDER --> REQ
    REQ --> HDR["headers · params : KV[]"]
    REQ --> BODY["body : RequestBody"]
    REQ --> AUTH["auth : RequestAuth"]
    REQ --> SCR["scripts.postResponse"]
    REQ --> EX["examples : SavedExample[]"]

TreeNode = FolderNode | RequestNode — a discriminated union on type. Folders nest recursively; every tree helper in lib/tree.ts walks this shape.

The pieces worth memorising

  • KV — the universal row: { id, key, value, enabled }. Headers, query params, and every kind of variable are all KV[]. The stable id is what stops React lists from remounting while you type.
  • RequestNode — a single request: method, url, headers, params, body, auth, scripts, saved examples. Its params are kept in two-way sync with the URL’s query string.
  • RequestAuth.mode — inherit means “use the environment’s token variable as a Bearer.” That one convention connects auth to the variable system.

Two models of a request: unresolved vs. resolved

This distinction trips people up, so pin it now. A RequestNode still contains {{variables}}. A RunRequest is the fully-resolved thing actually sent — headers flattened to [key, value] tuples, all variables substituted. The main process turns the former into the latter. The historical record of an execution is a Run (full request + response + script result); list views use the lighter RunSummary, projected by toSummary(run).

04 — CLIENT STATE

The state layer: three stores

No Redux, no React context. Three Zustand stores hold all client state; components subscribe to slices with selector hooks, and every user action calls a store action.

useApp

stores/app.ts

The workspace: collections tree, environments, settings, current selection, open tabs, and per-request drafts. Owns all persistence.

useRuns

stores/runs.ts

Execution + history: send(), the per-request run panel with diff-compare, and the global History page.

useUi

stores/ui.ts

Pure view state: current view, active editor tab, every modal flag, the find bar, context menu, toasts, inline-rename target. Never touches disk.

Persistence tiers

A subtle but important detail: different state saves on different schedules, all from useApp.

Open tabs are restored on boot from localStorage['relay:tabs:<workspaceId>'], filtered down to request ids that still exist. Sending a request never persists it.

05 — THE CENTRAL MECHANISM

Drafts vs. saved requests

If you understand only one internal mechanism, make it this one — it explains dirty-dots, unsaved edits surviving tab switches, and why you can send a request you haven’t saved.

There are two representations of any request: the saved node inside collections[].items, and an optional draft override in drafts[requestId] that exists only while there are unsaved edits. Every read goes through effectiveRequest() — “draft if present, else saved.”

flowchart TD
    E["User edits a field"] --> U["updateRequest(patch)"]
    U --> B["base = drafts[id] ?? saved"]
    B --> N["next = { ...base, ...patch }"]
    N --> URL{"url changed &
no explicit params?"}
    URL -->|yes| SYNC["re-derive params
from URL"]
    URL -->|no| EQ
    SYNC --> EQ{"next deep-equals
saved node?"}
    EQ -->|yes| DEL["delete draft
→ no longer dirty"]
    EQ -->|no| SET["drafts[id] = next
→ dirty-dot shows"]

The deep-equality check is what makes editing a value back to its original silently clear the dirty state — a small touch that feels right in the UI.

  • saveDraft() writes the draft into the collection tree via mapRequest, deletes the draft, and saves immediately.
  • discardDraft() just deletes the draft; the saved node reappears.
  • Drafts are keyed by request id, independent of tabs — so every open tab keeps its own unsaved edits and its own dirty-dot simultaneously.
  • Drafts are pruned in exactly one place: when a node or collection is deleted (pruneDrafts + pruneTabs).
06 — THE CROWN JEWEL

Sending a request, end to end

This is the flow that ties everything together. Follow one press of Send across the process boundary and back.

sequenceDiagram
    participant U as UI (Send btn / ⌘↵)
    participant RS as useRuns.send
    participant BR as window.relay
    participant H as http:send handler
    participant HC as sendHttp (fetch)
    participant SC as runScript (vm)
    participant D as disk + broadcast
    U->>RS: send()
    RS->>RS: effectiveRequest() → draft, unsaved OK
    RS->>BR: send({ sendId, collectionId, request })
    BR->>H: ipcRenderer.invoke(http:send)
    H->>H: merge vars, resolve auth + body → RunRequest
    H->>HC: sendHttp(sendId, resolved, limit)
    HC-->>H: { response, durationMs } or { error }
    H->>SC: runScript(postResponse, response)
    SC-->>H: { assertions, varsSet, logs }
    H->>D: write varsSet→env · appendRun · broadcast
    H-->>RS: full Run
    RS->>RS: re-pull env, prepend summary, set currentRun
    RS-->>U: ResponsePane renders

Step by step

  1. The renderer resolves which request to send. useRuns.send() reads the selection and calls effectiveRequest() — it sends the draft, so unsaved edits go out (Postman behavior). It mints a sendId for cancellation and flips sending: true.
  2. Across the bridge with the raw node. The SendPayload carries the unresolved RequestNode — {{vars}} still intact. All resolution happens in main, on purpose.
  3. Main builds the variable map. In register.ts the handler loads workspace/env/settings/collections in parallel, picks the active environment, and merges: { ...collectionVars, ...envVars } — environment variables win on a name clash.
  4. Resolve URL, headers, auth, body. Each is interpolated; disabled/blank rows are skipped. auth: 'inherit' + a token var becomes Authorization: Bearer …. The result is a fully-resolved RunRequest with no variables left.
  5. Actually make the request. sendHttp() in httpClient.ts uses Node’s global fetch, registers an AbortController in an inflight Map keyed by sendId, arms a 30s timeout, follows redirects, buffers the body as text, and truncates past the size limit (default 1 MiB) while still recording the true byte size.
  6. Run the post-response script. If present, runScript() executes the user’s JS in a Node vm sandbox (1s timeout) exposing res, vars.set/get, assert, console.log. Any vars.set(...) is captured in varsSet.
  7. Persist and notify. varsSet is written back into the active environment (this is how vars.set("chargeId", res.json.id) chains requests). The full Run is appended to today’s JSONL, and a RunSummary is broadcast to every window.
  8. The renderer catches up. send() re-pulls environments (a script may have changed them), sets currentRun, and prepends the summary into the run panel. ResponsePane re-renders with status, timing, size, assertions, and the pretty-printed body.

▲ Gotchas worth knowing

  • The 30s timeout is hardcoded, not a setting.
  • The full response is downloaded into memory (res.text()) before truncation — a giant response is fully fetched even though only 1 MiB is stored.
  • Redirects are opaque (redirect: 'follow'): you see only the final response, no chain.
  • The vm sandbox is crash/hang containment, not a security boundary — scripts are your own, Postman-tier trust. The code itself documents the upgrade path (an Electron utilityProcess) if that ever changes.
07 — WHY PREVIEWS MATCH SENDS

Variable interpolation, two phases

One engine in src/shared/interpolate.ts serves both the UI and the sender. It matches canonical {{name}} (and legacy ${name}), where names may start with $ for dynamic built-ins.

The clever bit is the dynamic flag:

  • Renderer preview calls it with dynamic: false — so {{$uuid}} and {{$timestamp}} are left literal in what you see.
  • Main, at send time uses dynamic: true — $uuid becomes a fresh crypto.randomUUID(), $timestamp becomes unix seconds.

The payoff: a dynamic value you preview never disagrees with the one actually sent, because the preview deliberately doesn’t evaluate it. Unresolved names are returned in an unresolved list, which drives the “missing variable” coloring in the URL bar and the var-peek popover.

Resolution precedence is assembled by mergedVars() in stores/app.ts — collection variables first, active-environment variables spread last so they override. The same merged map feeds URL highlighting, {{ autocomplete, hover peek, and code generation.

08 — WHERE BYTES LAND

Persistence on disk

Root is Electron’s userData (~/Library/Application Support/Rewind on macOS). Settings and window geometry are global; everything else is per-profile.

userData/
├─ settings.json            ← global
├─ window-state.json        ← global
├─ profiles.json            ← registry { activeId, profiles[] }
└─ profiles/
   └─ <profileId>/          ← the "active profile dir"
      ├─ workspace.json
      ├─ environments.json
      ├─ collections/
      │   └─ <collectionId>.json    ← one file per collection
      └─ runs/
          └─ YYYY-MM-DD.jsonl        ← append-only, one run per line

Three storage habits worth copying

  • Atomic writes (jsonStore.ts): write to a .tmp, fsync, then rename (atomic on APFS). Per-path write queues serialize concurrent writes to the same file so they never interleave.
  • Corrupt-file quarantine: a JSON file that fails to parse is renamed to .corrupt-<ts> and treated as absent — never deleted, never fatal. The JSONL parser likewise skips a torn trailing line from a mid-write crash.
  • Append-only run log with an mtime cache (runLog.ts): history lists are cheap because per-day files are only re-parsed when their mtime changes; full Run bodies are read on demand.

Profiles are almost free: a single mutable activeDir is repointed on switch, so all path helpers follow. initProfiles() also migrates any legacy root-level data into the first profile on first run.

09 — THE VISIBLE PART

The UI, top to bottom

App.tsx gates on boot, then renders the Titlebar and either the RunbookView or the HistoryPage, plus a stack of modals driven by useUi flags. The runbook is a three-column grid:

RunbookView
├─ Sidebar          collections tree, search, selection
├─ center-pane
│   ├─ TabsBar      open requests (Postman-style)
│   ├─ UrlBar       method · url · Send
│   ├─ RequestTabs  Params · Body · Headers · Auth · Scripts
│   └─ ResponsePane status · body · find · copy-as
└─ HistoryPanel     this request's recent runs + diff

How a click becomes state

The pattern is uniform: an interaction calls a store action, the store mutates, subscribed components re-render. Selecting a request in the Sidebar calls selectRequest(), which adds a tab and sets selection; a useEffect in RunbookView then fires loadForRequest() to fetch that request’s run history. Every editor field routes through updateRequest() — so all editing is draft-based and reversible until Save.

Two details you’ll re-encounter

  • Tabs are just openTabs: Selection[] plus one selection; the invariant is that selection is always one of the tabs or null. Middle-click or ✕ closes; close moves selection to the nearest neighbor.
  • Splitters (sidebar width, editor/response height) share one deliberate trick: during drag they write the size straight to the DOM per animation frame and only commit to settings on pointer-up — so a drag never re-renders the whole pane.
10 — THE REST OF THE MAP

Supporting subsystems

Grab-bag of the remaining machinery — you now have enough context for each to make sense at a glance.

  • Import pipelines. openapiImport.ts walks paths × methods, groups by tag into folders, and synthesizes body examples from JSON Schema. postmanImport.ts sniffs and converts every Postman shape (v1, v2.0, v2.1, environments, full data-dump), remaps variable syntax, and imports pm.* test scripts commented out since they can’t run in Rewind’s sandbox.
  • Code generation (shared/codegen.ts). buildCurl, buildNode, buildPython all consume an already-resolved RunRequest — so “Copy as cURL” is just resolveForCodegen() + one builder.
  • cURL paste (shared/curlParse.ts). A real quote/escape-aware tokenizer turns a pasted curl command back into a RequestNode, which is why pasting a URL re-syncs the params table.
  • Response diffing (lib/jsonDiff.ts). Normalizes both bodies (without reordering keys — order changes are real), line-diffs them, and merges adjacent changes down to a token boundary so 2000 → 2500 never renders as 20 → 500.
  • Syntax highlighting (lib/tokenize.ts). A single-pass tokenizer for JSON and JS powers the editor, the response viewer, and the diff — with plain-<pre> fallbacks past size thresholds for performance.
  • The seed (seed.ts). First run writes a demo “Payments API” workspace with realistic historical runs — including a Create charge request whose script does vars.set("chargeId", res.json.id), the canonical request-chaining example.
11 — DO THIS NEXT

A guided reading tour

Now open the code in this order — each step builds on the last, and you’ve already got the mental model for all of them.

  1. The contract. Read src/shared/types.ts top to bottom, then src/shared/ipc.ts. This is the whole vocabulary in ~250 lines.
  2. The bridge. Skim src/preload/index.ts — see how thin it is. Every method maps 1:1 to a channel.
  3. The state. Read stores/app.ts, focusing on updateRequest, effectiveRequest, saveDraft, and mergedVars. This is section 05 in code.
  4. The send path. Follow send() in stores/runs.ts → the http:send handler in ipc/register.ts → sendHttp in httpClient.ts → runScript in scriptRunner.ts. This is section 06 in code.
  5. The storage. Read jsonStore.ts and runLog.ts to see the crash-safe patterns.
  6. Pick one feature and trace it fully. Good candidates: a header edit (UI → updateRequest → draft → debounced save), or “Save example” (ResponsePane → runs:saveExample → collection rewrite).

◆ The one-sentence summary

A sandboxed React UI holds all editing in memory as drafts, resolves variables only for preview, and hands raw requests across a single typed bridge to a Node process that does the real HTTP, runs your scripts, and writes everything to local JSON — crash-safely, one file per collection and one JSONL line per run.
